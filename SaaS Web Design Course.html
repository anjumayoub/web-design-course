The rest of the foundation

Nielsen's 10 usability heuristics are the vocabulary of every design critique you'll ever be in. Read the canonical article and, for each heuristic, name one example from an app you use daily: 10 Usability Heuristics for User Interface Design (NN/g).

Information architecture — on web this becomes URL structure, nav hierarchy, and search. A SaaS app is a tree: workspaces → projects → items → detail. Users must always know where they are (breadcrumbs, highlighted nav, page titles answer this).

User flows — you know these from mobile. The web difference: flows can branch through multiple entry points (deep links, email links, shared URLs, SEO landings). A web screen never gets to assume "the user came from the previous screen."

Mental models — write down what the user believes is happening ("my draft is saved", "this filter is temporary"). Interfaces fail where the model and the system disagree. Autosave indicators, optimistic UI, and undo exist to close these gaps.

Week 1 · Lesson 1.2

Mobile → Web: everything that changes

This is the most important lesson in the course. Web isn't "a bigger phone" — it's a different medium with a different input device, different viewport physics, and different user posture.

1. The canvas is fluid, not fixed

Mobile gave you a handful of known devices. Web gives you a continuum from 320px phones to 3840px ultrawides — and users resize windows mid-session. You don't design "a screen"; you design a system of rules that produces the right screen at every width. That's why grids (Week 2) and responsive logic (Week 8) are core skills, not add-ons.

RealityMobileWeb
Viewport~390×844, fixed, portrait320→3840px wide, user-resizable
Design frameDevice frame1440×900 artboard + a max-width container (usually 1200–1280px) with rules for what stretches
OrientationPortrait-firstLandscape-first; vertical space is the scarce resource

2. The pointer changes everything

A finger is ~9mm of imprecise meat with no position until it touches. A cursor is 1px-precise and always present — which unlocks an entire layer mobile doesn't have: hover.

  • Hover states — every interactive element needs one (buttons, rows, cards, links). Hover is how web users probe "is this clickable?"
  • Hover-revealed actions — row actions in tables, card menus, comment buttons. Density without clutter. (Rule: hover-revealed must also be reachable by keyboard.)
  • Tooltips — hover explains icons and truncated text. There is no long-press.
  • Cursor changes — pointer for clickable, text for selectable, grab for draggable. Free affordance.
  • Smaller targets are OK — 32–40px buttons instead of 44pt minimums; 13–14px UI text instead of 16–17pt. This is where SaaS density comes from.

And the keyboard becomes a first-class input: focus states (visible rings on the focused element), Tab order, Enter to submit, Esc to close, and power-user shortcuts (Cmd+K command palettes). Every component you design now needs states mobile never asked for: defaulthoverfocusactivedisabledloading

3. Reading behavior: scanning patterns

On wide screens, eyes don't read — they scan. Two patterns you design for:

  • F-pattern — on text-heavy pages users read the first lines, then scan down the left edge. Front-load headings and list items with keywords; left-align what matters. Read: NN/g on the F-pattern.
  • Z-pattern — on sparse marketing layouts the eye sweeps top-left → top-right, diagonals down, ends bottom-right. Logo top-left, CTA top-right and again at the diagonal's end — the classic landing header follows the Z.

4. Scroll, chrome, and the fold

  • Scrolling is cheap, but the top ~600px ("above the fold") still decides whether users continue. Marketing pages must state value + CTA before any scrolling.
  • Browser chrome exists — the URL is UI (readable, shareable, deep-linkable). Back button must never break. New-tab behavior matters (marketing links may open new tabs; in-app navigation never).
  • Sticky elements — headers, table headers, sidebars, and in-page section navs can stick while content scrolls. Powerful and easy to overuse: every sticky element permanently taxes the scarce vertical space.

5. Posture and session shape

Mobile sessions are 30-second bursts on the go. SaaS web sessions are 30-minute-to-8-hour work sessions at a desk. Consequences: information density is a feature (fewer clicks > more whitespace), multitasking is normal (your app lives in one of 12 tabs — favicons, page titles and notification badges matter), and efficiency compounds (bulk actions, keyboard shortcuts, saved views: features that would be overkill on mobile are table stakes in SaaS).

Start doing
  • Design hover + focus states for every component
  • Design at 1440 with a 1200–1280px container
  • Treat vertical px as the scarce resource
  • Ask "what does the URL say?" for every screen
  • Add tooltips to every icon-only button
Stop doing (mobile habits)
  • 44pt touch targets everywhere
  • Bottom nav / thumb-zone thinking
  • Hiding actions behind long-press or swipe
  • One-thing-per-screen pacing for work tools
  • Assuming users arrived from the previous screen

Week 2 · Lesson 2.1

The 12-column grid, properly

The grid is the skeleton of every web page you'll ever design. Mobile mostly hid this from you — a phone is basically one column. On web, the grid is the design.

Watch

▶
How to Set Up a 12-Column Grid System in Figma
Follow along in Figma — set this up before reading on.

The vocabulary

TermMeaningTypical web values
ContainerThe max width your content lives in; centered, with side margins1200–1280px (marketing), fluid/full-width (apps)
ColumnsVertical slices content snaps to12 (marketing & dashboards), 8 (tablet), 4 (mobile)
GutterGap between columns24px (also 16, 20, 32)
MarginSpace between the grid and the viewport edge≥32px desktop, 24px tablet, 16px mobile

Why 12?

12 divides by 2, 3, 4 and 6 — so one grid supports halves, thirds, quarters and sixths without changing anything. A pricing page with 3 plans = three 4-column blocks. A blog with sidebar = 8 + 4. A form next to an illustration = 6 + 6. Every layout decision becomes "how many columns does this get?" — which is exactly the kind of constraint that makes design faster.

Your default Figma setup (memorize this)

  • Frame: 1440×900
  • Layout grid: 12 columns, Stretch, gutter 24, margin 120 → yields a 1200px content container
  • Alternative for denser marketing: margin 80 → 1280px container
  • Add a second grid: Rows, 8px (10% opacity) — this is your spacing ruler (next lesson)
Stretch vs Fixed columnsSet columns to Stretch, not fixed-width. Stretch mirrors how CSS grids actually behave when the viewport changes — your design intent survives translation to code. Fixed-width columns are a mobile-designer habit that breaks responsive thinking.

Apps vs marketing pages use grids differently

  • Marketing pages — content sits inside the centered container; the 12 columns govern every section. Backgrounds can bleed full-width while content stays in the grid ("full-bleed section, contained content" — look at any Stripe page).
  • SaaS apps — the app shell (sidebar + top bar) is not on the 12-col grid; it's a fixed-width sidebar (240–280px) plus a fluid content region. Inside that content region you re-apply a grid or a card system. Dashboards often use 12 columns inside the content area for arranging widgets.

Week 2 · Lesson 2.2

Spacing systems & vertical rhythm

Columns handle horizontal structure. Vertical structure comes from a spacing scale — and consistent spacing is the single most visible difference between amateur and professional web design.

The 8-point system

All spacing (padding, gaps, margins) comes from multiples of 8: 8, 16, 24, 32, 40, 48, 64, 80, 96, 120, 160 — with 4 and 12 allowed for fine-grained UI (icon gaps, badge padding, dense tables). Why it works: it's divisible, it renders crisply at every screen density, and it kills the "is this 27 or 29px?" micro-decisions that eat your day.

Spacing communicates grouping (this is the real skill)

Proximity is hierarchy. A label sits 8px from its input, but 24px from the previous field — so the eye pairs them correctly without a single divider line. The general rule: space within a group < space between groups < space between sections. When a design feels "off" and you can't say why, it's almost always spacing that contradicts grouping.

RelationshipTypical value
Icon ↔ its label4–8
Label ↔ its input6–8
Between form fields16–24
Card internal padding16–24 (dense UI) / 24–32 (marketing)
Between cards16–24
Between page sections (app)32–48
Between page sections (marketing)96–160
Marketing spacing will feel wrong at firstMobile trained you on tight vertical budgets. Marketing sections breathe with 96–160px between them. When your landing page feels cramped, double the section spacing — seriously, double it — before touching anything else.

White space is doing work

It buys hierarchy without adding lines or boxes (Refactoring UI's "start with too much white space, then remove"), it signals quality (luxury = space; discount = cramming — think Apple vs a flight-deals site), and it sets reading pace. Dense SaaS tables minimize it deliberately; long-form content maximizes it deliberately. Density is a decision, not an accident.

Vertical rhythm

Keep vertical gaps from the same scale so repeated patterns (cards in a grid, rows in a list, sections down a page) create a steady beat. Turn on your 8px row grid and nudge until things land on it. Don't be religious — optical alignment beats mathematical alignment when they conflict (more in Week 3).

Web layout patterns that repeat forever

Almost every web page you'll ever design is a remix of about a dozen layouts. Learn them by name and you'll never face a blank canvas again.

PatternStructureUsed for

Section-level patterns (marketing pages are stacks of these)

  • Hero — headline + subhead + CTA(s) + visual. Variants: centered (text over/above visual), split (text left, product right), full-bleed background.
  • Logo bar — "Trusted by" strip of 5–8 customer logos, greyscaled to not compete.
  • Feature trio — 3 columns of icon + title + 2 lines. The workhorse.
  • Zig-zag — alternating image/text rows (image left, then image right…). Rhythm for long feature lists.
  • Bento grid — asymmetric card mosaic showing many features at once; the current-era pattern (Apple, Vercel, Linear all use it).
  • Social proof — testimonial cards, case-study stats ("300% increase in…"), review badges.
  • Pricing — 3 tiers, middle highlighted (Week 7 goes deep).
  • Final CTA — big headline + one button on colored background, right before the footer.
  • Footer — 4–5 link columns + legal row. Mega-footers double as sitemaps and SEO.

Visual hierarchy on wide screens

Three levers, in order of power: size (biggest thing wins — one per screen), contrast (dark text/strong color pulls focus; muted grays recede), position (top-left of the F-pattern; the Z's endpoints). If everything is big and bold, nothing is — pick winners deliberately. A good self-test: squint until the page blurs. Whatever still stands out is your actual hierarchy; check it matches your intended one.

Do
  • Pick an archetype before opening Figma
  • Limit line length even when the screen is wide
  • Let sections alternate background tint (white / gray-50) to separate them
  • Anchor every element to the column grid
Don't
  • Stretch content edge-to-edge on 1920px screens
  • Center-align long body text
  • Invent a new layout when a named one fits
  • Fill horizontal space just because it exists

Week 3 · Lesson 3.1

Choosing & pairing typefaces for web

On mobile you mostly inherited SF Pro or Roboto. On web, typeface choice is yours — it sets 70% of a site's personality before a single color is applied.

The categories and what they signal

CategorySignalsWeb examples
Neo-grotesque sansNeutral, modern, "product"Inter, Helvetica Now, SF Pro, Roboto

Notice the current SaaS pattern: a neutral sans for UI (Inter is everywhere) + one personality font for marketing headlines (a display serif or distinctive sans). Origin's landing page from Week 2 does exactly this — serif headline, sans everything else.

Rules for choosing UI type

  • Tall x-height, open apertures — legible at 13px. This is why Inter, not Futura, runs your dashboard.
  • Real weights — you need at least 400 / 500 / 600. (Faux-bolding is a crime the browser will happily commit for you.)
  • Tabular figures available — numbers that align in columns. Non-negotiable for tables and dashboards (Week 6).
  • Variable font versions — one file, all weights, faster loads. Inter, Fraunces, Source families all ship variable.

Pairing without pain

  • The safest pair is no pair — one family, many weights/sizes. Most great SaaS products are single-typeface.
  • Pair by contrast, not similarity — display serif + neutral sans works; two similar sans faces just looks like a mistake.
  • One does headlines, one does everything else. Never alternate mid-hierarchy.
  • Steal proven pairs: Google Fonts Knowledge and the specimen pages on fonts.google.com show live pairings.

Web-only concerns mobile never taught you

  • Loading — webfonts download. Fewer families/weights = faster paint. System-font stacks cost 0ms (GitHub uses one).
  • Fallbacks — specify what shows if the font fails: Inter, -apple-system, sans-serif. Design shouldn't collapse on fallback.
  • Licensing — Google Fonts = free anywhere. Foundry fonts (Circular, Canela…) need web licenses priced by pageview. Check before you fall in love.

Week 3 · Lesson 3.2

Setting type: scale, rhythm, measure

Choosing a font is 10% of typography. The other 90% is setting it: sizes, line-heights, line lengths, and the relationships between them.

Units: px, rem, em

You design in px in Figma, but developers build in rem (1rem = 16px by default, scales when users change browser font size). Keep every font size divisible into sensible rems (13, 14, 16, 18, 20, 24…) and note rem equivalents in handoff. em is relative to the parent element — used for things that should scale with their context, like an icon inside a button.

Type scales

Pick sizes from a ratio, not from vibes. Generate one at typescale.com:

RatioFeelBest for
1.125 (major second)Subtle stepsDense app UI
1.2 (minor third)BalancedThe default choice — apps + content
1.25 (major third)Confident jumpsMarketing pages
1.333–1.5DramaticEditorial, big-brand landings

A practical dual-scale setup for a SaaS product: App UI: 12 · 13 · 14 (body) · 16 · 18 · 20 · 24. Marketing: 16 · 18 (body) · 20 · 24 · 32 · 40 · 56 · 72. Marketing headlines go far bigger than mobile ever allowed — a 64px H1 at 1440 is normal, not shouting.

Line height (leading)

TextLine heightExample
Big headlines (40px+)1.0–1.1556px / 60px
Subheads (20–32px)1.2–1.3524px / 32px
Body (14–18px)1.5–1.716px / 26px
Dense UI text (12–14px)1.35–1.513px / 18px

The inverse law: the bigger the text, the tighter the leading. Mobile rarely showed you 56px headlines, so this is a new muscle — a 56px headline at 1.5 line-height looks broken.

Measure (line length) — the most-violated web rule

45–75 characters per line; ~66 is ideal. At 16px that's roughly a 640–700px column. This is why article text sits in a narrow centered column even on a 1920px monitor — and why "stretch the text to fill the space" is always wrong. Wide lines make eyes lose their place on the return sweep; short lines create choppy, fatiguing rhythm.

Letter-spacing (tracking)

  • Big headlines: slightly negative (-1% to -3%) — large glyphs look loose by default. This is the "designed" look of modern SaaS headlines.
  • Body: leave it alone (0).
  • ALL-CAPS labels: positive (+4% to +8%) — caps need air. (Like the section labels in this course's sidebar.)

Hierarchy without size

Before making something bigger, try: heavier weight, darker/lighter color (ink vs muted gray), or more surrounding space. Refactoring UI's trick: a hierarchy of three text colors (primary #0F172A-ish, secondary #64748B-ish, disabled/hint #94A3B8-ish) does more work than five font sizes. Size changes are the loudest tool — save them for real structure.

Do
  • Define the full scale before designing screens
  • Keep body text 16–18px on marketing, 13–14px in dense UI
  • Cap text columns at ~70ch regardless of screen width
  • Tighten tracking on 40px+ headlines
Don't
  • Invent a new size per screen ("size soup")
  • Use pure black text on pure white (harsh; use near-black)
  • Center-align paragraphs longer than 2 lines
  • Justify text on the web (rivers everywhere)

User-generated text will be longer than your mock. Building a color system

Web products don't use colors — they use color systems: structured ramps of shades that make every future decision automatic.

Watch

▶
Super Practical Guide to Color Theory, Color Models & Perfect Palettes

Think in HSL, not hex

Hex codes are for computers. Design in Hue (0–360° on the wheel), Saturation (gray→vivid), Lightness (black→white). Every skill in this lesson is an HSL manipulation: shades of a color = same hue, stepped lightness (with saturation adjusted); a "warmer gray" = a gray with 2–5% saturation of an orange hue; a harmonious palette = controlled hue relationships.

What a real UI palette contains

GroupContentsJob
Primary/brand1 hue × 9–10 shades (50→900)Buttons, links, active states, focus rings
NeutralsGray ramp × 10 shades — where 90% of your UI livesText (3 levels), backgrounds, borders, disabled
SemanticGreen (success), red (danger), amber (warning), blue (info) — each with a pale background shadeFeedback: alerts, toasts, validation, badges
Accents (optional)1–2 supporting hues, data-viz rampCharts, tags, illustration, marketing moments

Build ramps like the pros: pick your brand color as the ~500 shade, then step lightness both ways — but desaturate slightly toward white and shift saturation/hue subtly toward the extremes, or the pale shades look washed-out and the dark ones dead. Compare your handmade ramp against Tailwind's palette — the most battle-tested open ramp set on the web (and the palette this course file uses).

Proportions: 60-30-10, translated for UI

In practice: ~60% neutral backgrounds, ~30% neutral text/borders/surfaces, ~10% brand color — and semantic colors only at moments of feedback. New web designers' most common mistake is 10× too much brand color. Open Linear or Stripe's dashboard: they're gray products where color means something is interactive or something happened.

Saturated colors are for small areasA full-saturation indigo works at button scale, but as a page background it exhausts. Big areas → very pale tints (your gray-50/indigo-50); high saturation → small, meaningful moments.

Choosing the brand hue

  • Blue/indigo/violet: trust, tech, the SaaS default (for a reason — safest, and renders well on screens)
  • Green: money, health, growth, "go"
  • Warm hues (orange/coral): energy, consumer-friendly, creative tools
  • Black/near-black as brand: premium, developer-brands (Vercel, Linear's marketing)
  • Check the niche's conventions on Mobbin first — fintech skews blue/green + dark; creative tools go bolder. Differ deliberately, not accidentally.

Week 4 · Lesson 4.2

Color in practice: contrast, tokens, dark mode

A palette is potential. This lesson is application: making color readable, systematic, and theme-able.

Contrast — the non-negotiable

WCAG contrast ratios run 1:1 (identical) to 21:1 (black on white). The AA standard you design to:

ElementMinimum ratio
Body text (<24px)4.5:1
Large text (≥24px, or ≥18.7px bold)3:1
UI components & meaningful icons/borders3:1

Check every text/background pair with WebAIM's checker or a Figma plugin (Stark, Contrast). The classic failures: muted gray helper text on white (your gray-400 probably fails — use gray-500+), white text on brand-500 buttons (often needs brand-600), and placeholder text nobody can read.

Color is never the only signalColorblind users (~8% of men) can't rely on red-vs-green. Pair color with an icon, label, or pattern: error states get an icon + message, not just a red border; chart series get distinct shapes or direct labels, not just hues.

Text on colored & image backgrounds

  • On brand-colored surfaces, white text usually needs the surface to be ≥500-level dark. Check it — don't eyeball it.
  • On photos: add a scrim (dark gradient overlay), blur the image region, or anchor text to a solid panel. Never raw white-on-photo.
  • "Colored text on colored bg" (like this course's callout boxes): use the 700–800 shade for text on the 50 shade background of the same hue.

Semantic tokens — naming color by job

Raw shades (indigo-600) are primitives. Products map them to semantic tokens named for their role: bg/surface, bg/subtle, text/primary, text/secondary, border/default, action/primary, action/primary-hover, feedback/danger… Components reference tokens, tokens reference primitives. Change what action/primary points to and the whole product rebrands. This two-layer structure is exactly what Figma Variables model (Week 8 builds it).

Dark mode basics (SaaS table stakes)

  • Not inverted, re-mapped: dark surfaces are dark-gray (slate 900/950), never pure black; elevation = slightly lighter surface, since shadows die in the dark.
  • Desaturate: vivid brand colors vibrate on dark — use the 400 shade where light mode used 600.
  • Soften text: pure white glows; use gray-100/200 for primary text. Contrast rules still apply, same math.
  • Semantic tokens make dark mode a re-mapping exercise instead of a redesign — the reason the token layer exists.

Elevation: a shadow scale, not random shadows

Define 4–5 elevation levels and use them consistently — shadows communicate what floats above what:

Borders vs shadows vs background shifts

Three ways to separate surfaces: borders (crisp, dense-UI friendly — use gray-200-ish, 1px), shadows (soft, floaty, marketing-friendly), background contrast (white cards on gray-50 page — no outline needed at all). Week 5 · Lesson 5.1

The component encyclopedia

Every web interface is assembled from ~25 recurring components. You know many from mobile — but each carries web-specific rules, states, and conventions. This is your reference lesson; return to it often.

Buttons

  • Hierarchy: primary (filled, brand — one per view), secondary (outline/tonal), tertiary/ghost (text-only), destructive (red, for irreversible actions), icon-button (needs tooltip).
  • Sizes: sm 28–32px, md 36–40px (default), lg 44–48px (marketing CTAs). Padding ≈ 2× vertical padding horizontally.
  • All six states from Week 1 — hover darkens/lifts, focus shows a ring, loading swaps label for spinner without changing width.
  • Labels: verbs ("Save changes", "Create project"), never "Yes"/"OK" alone. In dialogs, buttons state the outcome: "Delete 14 items" beats "Confirm".

Inputs & selection controls

  • Text input: label above (not placeholder-as-label), 36–44px tall, visible border (3:1 contrast), focus ring, error state with icon + message below, optional helper text. Prefix/suffix slots ($ 0.00 USD).
  • Select vs radio vs combobox: ≤5 options visible → radio group; 6–15 → select dropdown; >15 or user-known values → combobox with type-ahead search.
  • Checkbox = independent toggles / multi-select; radio = exactly one; switch = instant on/off with immediate effect (no save button); if a save button exists, use checkboxes, not switches.
  • Specialized: date pickers (allow typing too — pickers alone infuriate), steppers, sliders (only for imprecise ranges), file dropzones, tag inputs.

Overlays

ComponentUse whenWeb rules
ModalFocused task or confirmation that must interruptMax ~480–640px wide, close via X + Esc + (usually) backdrop click, never stack modals, primary action bottom-right
Drawer/panelDetail view or editing without losing list contextSlides from right, 360–480px; the SaaS favorite for record details
PopoverSmall contextual UI anchored to a trigger (filters, pickers)Dismiss on outside click; keep within viewport
TooltipName/explain on hover — icons, truncations, shortcutsAppears on hover+focus, ~300ms delay, never contains actions
ToastAsync feedback that shouldn't block ("Saved", "Export ready")Top/bottom-right, auto-dismiss 4–6s, include Undo where possible
The modal testBefore designing a modal ask: could this be inline, a drawer, or a dedicated page? Modals that contain whole workflows (scrolling forms, tabs!) are the most common SaaS design smell. Interrupt only to confirm or capture one focused thing.

Structure & wayfinding

  • Cards: a bounded surface = one unit of content. Don't nest cards; don't card everything — tables beat card-grids for comparing records.
  • Tabs: peer views of one object (Profile / Security / Notifications). Selected tab must be obvious; never tabs inside tabs.
  • Accordion: progressive disclosure for long optional content (FAQs, advanced settings). Not for content everyone needs.
  • Breadcrumbs: hierarchy path on deep structures (Workspace / Project / Issue). Marketing sites skip them; docs and consoles need them.
  • Pagination vs infinite scroll: data/work UIs paginate (position, linkability, footer reachable); feeds scroll infinitely. Show "1–50 of 1,204".
  • Badges/tags: status dots + label (colorblind-safe), count badges, removable filter chips.
  • Avatars: 24/32/40px, initials fallback with deterministic color, stacked "+4" groups.
  • Skeletons > spinners for initial loads (perceived speed); spinners for short unknown waits; progress bars when duration is estimable.

Sidebar anatomy (memorize)

Top: workspace/org switcher + search. Middle: primary destinations (icon + label, groups of 4–6 with labeled dividers — Miller's Law), then pinned/favorite items, then dynamic lists (projects, channels). Bottom: settings, help, user account. Active item gets a filled background; hover gets a lighter one; groups collapse. Look at this course's sidebar — same skeleton.

Universal SaaS furniture

  • Command palette (Cmd+K): search + actions + navigation in one. Now expected in professional tools; design it early, it shapes IA.
  • Global search: scoped results (projects vs docs vs people), recent items on focus, keyboard navigation.
  • Breadcrumb header in the content area: where am I + page-level actions on the right (the "page header" pattern: title left, primary button right).
  • User menu: profile, theme toggle, keyboard shortcuts, sign out. Org switcher separate from user menu.
  • Notifications: bell + badge count, popover inbox with read/unread, mark-all-read. Deep-links into the app (multiple entry points again).

Navigation depth rules

  • Primary nav shows place, not actions. "Create new" is a button, not a nav item.
  • Two levels max in a sidebar; a third level lives inside the page (tabs).
  • The URL mirrors nav depth: /projects/website-redesign/settings. If you can't express a screen as a URL, your IA has a hole.
  • Persistent selected states everywhere — users glance at nav to answer "where am I" dozens of times per session.

Forms that don't lose users

Forms are where products make money and lose users. Signup, checkout, settings, record creation — all forms. This is the highest-ROI component skill on the web.

Watch

▶
Multi-Step Form UX — Best Practices

Layout laws

  • Single column. The most replicated form finding: one column completes faster with fewer errors. Exceptions only for tightly-coupled pairs (First/Last name, City/Postcode, MM/YY · CVC).
  • Labels above fields, left-aligned. (Left-of-field labels are for dense enterprise settings pages only.)
  • Field width hints expected content: a postcode field shouldn't be 600px wide. Cap the whole form at ~400–480px.
  • Group with headings + spacing (Week 2's proximity rule), not boxes-inside-boxes.
  • One primary button, left-aligned with the fields (or full-width in narrow auth forms), labeled with the outcome: "Create account", "Pay $49".

Reducing friction (the real design work)

  • Cut fields ruthlessly. Every field costs conversion. No "fax number". Ask: could we get this later, infer it, or live without it? ("Progressive profiling" = ask later, in context.)
  • Smart defaults + inference: country from IP, city from postcode, name parsed from email. Tesler's Law applied.
  • Show, never punish: input masks/formatting as they type (card numbers grouping into 4s), password requirements listed upfront with live checkmarks, caps-lock warnings.
  • Optional is labeled, required is not (when most fields are required). "(optional)" beats asterisk soup.

Validation & errors

  • Validate on blur (leaving a field), not on every keystroke (premature reds while typing) and not only on submit (error avalanche). Exception: live-validate positively — turning requirements green as they're met.
  • Error anatomy: red border + icon + specific message below the field. "Password needs 8+ characters" not "Invalid input". Never color alone.
  • On failed submit: scroll to the first error, keep everything the user typed, summarize errors at top for long forms.
  • Never clear the form. Ever.

Multi-step forms (wizards)

  • Split by mental topic (Your details → Company → Preferences), 3–5 steps, one topic per step.
  • Progress indicator with named steps; allow going back without data loss; autosave state.
  • Start with the easy/engaging question, ask the sensitive stuff (phone, billing) last — commitment escalates.
  • Final step = review + edit links. The Coda/ClickUp onboarding flows you'll study in Week 7 are exactly this pattern.

The five states of every screen

Mockups show the ideal state. Real products live in the other four. Designing all five is what separates product designers from screen painters.

The five

StateDesign job
1. EmptyTeach + activate: explain what will live here, give the primary action, consider a template/sample-data shortcut. First-run empty ≠ cleared-out empty ≠ no-search-results — three different messages.
2. LoadingSkeletons matching final layout (no layout shift); per-widget, not per-page; optimistic UI for user actions (show the change instantly, reconcile after).

5. IdealThe one you were already designing — now with realistic data: long names, 0 and 9-digit numbers, missing avatars.

The pattern: illustration (optional), one line of value ("Reports show you where your time goes"), one primary CTA ("Create your first report"), one escape hatch ("Import from CSV" / "Use sample data"). Week 7 · Lesson 7.1

Auth & onboarding flows

The first five minutes decide whether a signup becomes a user. This is flow design: sequences, not screens.

Auth pages (the easy part)

  • Signup: email + password + SSO buttons (Google first — it converts best), or passwordless magic-link. Split layout (form + brand panel) or minimal centered card. Ask for nothing else — every extra field costs signups; get the rest during onboarding.
  • Login: same layout for muscle memory, plus "Forgot password" (never hide it), and clear error handling that doesn't reveal whether the email exists.
  • Password reset, verify-email, expired-link pages: boring, essential, usually forgotten by designers — so design them.

Onboarding = the path to the aha moment

Activation isn't account creation — it's the first moment of real value (first report run, first message sent, first design shared). Everything between signup and that moment is onboarding, and every step is a leak. The standard SaaS sequence:

  1. Profiling questions (1 per screen, 3–5 max): role, team size, intent — used to personalize the workspace and route marketing. Progress bar + Skip on every one.
  2. Workspace setup: name it (pre-filled from their name — smart defaults!), invite teammates (skippable, but this is the growth loop — make it attractive).
  3. The shortcut to value: templates, sample data, or an import — never a blank screen. "Start with a template" beats a tutorial every time.
  4. In-product guidance: a getting-started checklist (3–5 tasks with progress — the psychology of the endowed progress effect), plus contextual tooltips. Full product tours get skipped; checklists get finished.

Week 7 · Lesson 7.2

Core SaaS surfaces: settings, billing, teams

Every SaaS product ships the same five unglamorous surfaces. Design them by convention — users visit rarely, so familiarity beats cleverness (Jakob's Law at maximum strength).

Settings

  • Architecture: a settings sidebar grouped into Account (profile, security, notifications) vs Workspace/Org (general, members, billing, integrations) — this personal/organizational split is universal; blurring it confuses everyone.
  • Page pattern: single narrow column (~640px) of sections; each section = heading + fields; either autosave with "Saved" feedback per control (modern) or a save bar per section (classic) — never one giant save button for the whole page.
  • Danger zone: destructive actions (delete workspace, transfer ownership) live at the bottom, visually separated in red, with type-to-confirm dialogs.

Landing page = argument structure

A landing page is a persuasive essay in sections (you learned the section patterns in Week 2.3 — now order them as an argument):

  1. Hero: headline states the outcome you buy, in customer words ("Ship websites faster", not "AI-powered workflow platform"). Subhead = how, one CTA (+ optional secondary "watch demo"), visual proof (product shot).
  2. Credibility: logo bar / star ratings — borrow trust immediately.
  3. Problem → solution: name their pain, show the fix — feature sections framed as benefits ("Know where every rupee goes" over "Expense tracking module").
  4. Proof: testimonials with faces + names + companies, case-study numbers ("cut onboarding time 60%").
  5. Objection handling: FAQ, security badges, integrations wall, "works with your stack".
  6. Final CTA: restate the outcome + one button. Then footer.

Rules of thumb: one message per section; the CTA appears every ~2 screens of scroll; every section passes the squint test with its single takeaway; benefit-first headlines with features as supporting detail.

Pricing pages (the highest-stakes page)

  • Three tiers, middle highlighted ("Most popular") — anchoring + Hick's + Von Restorff in one pattern. Enterprise = "Contact sales".
  • Monthly/annual toggle with the annual discount framed as savings ("2 months free").
  • Per-tier: name, price with clear unit (per user/month, billed annually — the fine print that builds or breaks trust), one-line who-it's-for, feature list as differences (not 40 identical checkmarks), CTA per tier ("Start free" vs "Start trial" vs "Talk to us").
  • Below: full comparison table (sticky column headers!), FAQ handling billing objections.
  • Free trial vs freemium vs demo-first is a business decision that reshapes the whole page — know which model the product runs before designing.

Week 7 · Lesson 7.4

Beyond SaaS: e-commerce, content & portfolio sites

You asked for all of web design — this is the lesson that maps the rest of the territory. Same foundations (grid, type, color, components), different page anatomies and success metrics per niche.

E-commerce

The conversion funnel is the site: Home → Listing (PLP) → Product (PDP) → Cart → Checkout. Design each for its single job:

  • PLP (listing/category): filter sidebar (price, size, brand — with result counts), sort control, product cards (image dominates, name, price, rating, quick-add), grid of 3–4 columns. Filters are the product-finding UX — treat them as seriously as SaaS table filters (they're the same skill).
  • PDP (product page): gallery left (thumbnails + zoom), buy-box right: title, price (+strikethrough anchor), variant selectors (size/color as swatches — with availability states), quantity, Add to cart (the page's one hero button), shipping/returns reassurance right under it, then reviews, details accordion, cross-sell ("pairs well with").

  • Checkout: a multi-step form (Week 5.3 rules at maximum stakes): guest checkout first-class, address autocomplete, express pays (Apple/Google Pay) up top, order summary persistent in a right rail, no nav escape routes (the one place stripping navigation is correct).

Fluid + breakpoints (how the web actually resizes)

Layouts are fluid by default (percentages, max-widths) and break at defined widths where the arrangement changes. Content determines breakpoints — you add one where the layout starts failing, not at magic device numbers. The working set from Week 2 (390 / 768 / 1280 / 1440) covers real projects; design at 2–3 of them and specify the rules between.

Direction: which end do you start from?

  • Mobile-first — marketing, e-commerce, content, local business: most traffic is phones, and constraint-first forces message clarity.
  • Desktop-first — SaaS apps: work happens at desks; mobile web gets a companion subset (check status, approve, comment — not full admin).
  • Either way, you deliver both. "Is this page usable at 390?" is a question you now ask of every design, including dashboards.

The transformation playbook

PatternDesktop → Mobile
Top navLinks collapse to hamburger sheet/drawer; CTA button survives outside the menu
App sidebarIcon rail (tablet) → hidden drawer or bottom tabs (phone)
Card grid4 → 2 → 1 columns; the grid's column count is the dial
Split heroStacks: text above visual; center-align is acceptable here
Multi-column form pairsSingle column, always
ModalFull-screen sheet
Pricing trioStack with "most popular" first, or horizontal snap-scroll
Dashboard widgetsSingle column, KPI cards 2-up, charts simplified (fewer ticks, bigger touch targets)

Responsive tables (the boss fight)

Tables can't just squish. Choose per table, by how it's used:

  • Priority columns — keep identity + 1–2 key columns, move the rest into the row's detail view. (Most common, usually right.)
  • Horizontal scroll — freeze the identity column, let the rest pan. Honest for truly tabular data; signal scrollability with a cut-off column.
  • Card transform — each row becomes a stacked card (label: value pairs). Good for browse-y lists, verbose for dense data.
  • Skip the anti-pattern: shrinking font until it "fits". 9px type is not a strategy.

Also: touch replaces hover on mobile — every hover-revealed action needs a visible or menu-based fallback; and remember fluid type (clamp()) lets marketing headlines scale smoothly between breakpoints — specify min/max sizes in handoff.

Week 8 · Lesson 8.2

Accessibility that ships

Accessibility isn't a compliance checkbox — it's designing for the full range of human bodies and situations. It also makes everything better for everyone (captions, contrast, keyboard speed).

Watch

▶
WCAG 2.1 Crash Course

WCAG in one paragraph

The Web Content Accessibility Guidelines organize success criteria under POUR: Perceivable (can they sense it — contrast, alt text, captions), Operable (can they use it — keyboard, targets, timing), Understandable (can they follow it — plain language, predictable behavior, clear errors), Robust (does it work with assistive tech). Levels A → AA → AAA; AA is the professional and legal standard you design to.

The designer's checklist (what YOU control)

  • Contrast: 4.5:1 body text, 3:1 large text and UI components (Week 4 — you already do this)
  • Color never carries meaning alone: icon+label on statuses, underline-on-hover isn't enough for in-text links
  • Visible focus states on every interactive element — design them, don't let the browser default get stripped
  • Logical tab/reading order noted in specs; keyboard path for every hover-revealed action
  • Touch/click targets ≥ 24×24px minimum, 44px comfortable on touch
  • Real labels on all form fields + specific error messages (Week 5 — already yours)
  • Alt-text intent written for meaningful images; decorative images marked decorative
  • Text resizes to 200% without breaking — another reason for rem-friendly type
  • Motion: no autoplaying carousels without pause; respect reduced-motion preferences for big animations
  • Heading hierarchy is semantic (one H1, ordered levels) — screen readers navigate by it

Make it real, this week

Unplug your mouse for ten minutes and Tab through a site you admire — feel what keyboard users feel. Run WAVE on any page and read its complaints. Then audit one of YOUR Week 6–7 screens against the checklist above and fix what fails. Accessibility learned by audit sticks; accessibility learned by reading evaporates.

Design systems, tokens & handoff

Everything you've built — type styles, color primitives + semantics, spacing scale, component kit — has been a design system growing in secret. Now formalize it and learn to hand it to engineers.

The token pyramid

Primitives (indigo/600, space/4, radius/md) → semantic tokens (action/primary, text/secondary, bg/surface) → component tokens where needed (button/primary/bg). Components consume semantics, never raw values. In Figma this is Variables with modes: a light/dark mode pair on your semantic collection re-themes every screen instantly — the Week 4 promise, delivered.

What a shippable system contains

  • Foundations: color (primitives + semantic, both modes), the dual type scale, spacing scale, radius + elevation scales, grid definitions, icon set.
  • Components with variants covering real states — plus usage documentation: when to use, when not to, content rules ("buttons start with verbs"), a11y notes. Steal the documentation format from Polaris or Atlassian.
  • Patterns: the composed recipes — page header, settings section, table toolbar, empty state, dialog. Systems that stop at atoms leave the hard 50% undone.

Handoff that engineers respect

  • Name layers and components properly; use auto layout everywhere (it translates to flexbox nearly 1:1 — your padding/gap values ARE the spec).
  • Spec the invisible: hover/focus/loading states linked next to the default, responsive behavior per breakpoint ("sidebar → drawer below 1024"), motion (what animates, duration 150–250ms, easing), scroll behavior (what sticks), and every state of every flow (Week 6.3's five states).
  • Content edge cases in the file: longest name, empty avatar, 0 items, 12,847 items.
  • Sit in on implementation. The web designers engineers love are the ones who understand the box model, flexbox, and why "just move it 3px" isn't a thing. (Kevin Powell's channel keeps paying off here.)
